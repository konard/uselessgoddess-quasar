# quasar — проект и расчёты

Документ объясняет, почему модель устроена именно так, и во что она обходится.
Все параметрические числа здесь получены командой `quasar budget <preset>` и
сверены с реальным модулем тестом `model::tests::analytic_budget_matches_the_real_module`;
там, где число — оценка, а не измерение, это сказано явно.

## 1. Ограничения

| Ограничение | Значение | Что из него следует |
| --- | --- | --- |
| GPU | Radeon RX 9070 XT, 16 GB GDDR6, ~645 GB/s | VRAM — главный лимит, а не FLOPs |
| Backend | burn `main` (`d028234e`) + wgpu | нет AMP/autocast, нет кастомных ядер |
| `tiny` | 100–200M параметров, обучение ≤ 24 ч | ~1–3B токенов, см. §4 |
| `base` | 1–1.5B параметров, обучение неделями | ~3–8B токенов, см. §4 |
| Предшественник | [quark](https://github.com/uselessgoddess/quark), маленький трансформер, ppl ≈ 20 | нижняя планка качества |

Важное следствие первой строки: на этой карте **дефицитен не compute, а память и
пропускная способность**. Любая техника, которая меняет память на FLOPs, здесь
выигрывает; любая, которая делает наоборот (в первую очередь sparse MoE), —
проигрывает. Этим объясняется большая часть решений ниже.

## 2. Архитектура

Стек: `embedding → N × (norm → mixer → norm → SwiGLU) → norm → head`,
где каждый `attn_period`-й слой — GQA со скользящим окном, остальные — Mamba-3.

### 2.1 Почему Mamba-3, а не трансформер

Mamba-3 (2025) даёт три вещи, которых не было в Mamba-2:

- **трапецеидальная дискретизация** вместо метода Эйлера — тот же рекуррентный
  проход, но второй порядок точности; на равном бюджете это чистый выигрыш в
  качестве без FLOPs;
- **комплексный SSM**, эквивалентный data-dependent RoPE на `B`/`C` — возвращает
  способность к state tracking (parity, modular counting), которую вещественный
  SSM теряет структурно;
- **MIMO** — несколько параллельных каналов чтения/записи в одно и то же
  состояние. Это ключевой пункт для нашей карты: MIMO поднимает арифметическую
  интенсивность рекуррентности **не расширяя состояние**, то есть покупает
  качество за FLOPs, которых у нас относительный избыток, а не за память.

Короткая свёртка (`conv1d`), обязательная в Mamba-1/2, в Mamba-3 не нужна —
её роль берут на себя RoPE-компоненты. Это минус один memory-bound kernel в
слое, что на wgpu заметно.

В конфиге: `state_rank = 128`, `head_dim = 64`, `rope_fraction = 0.5`
(половина измерений состояния несёт вращение — именно эта доля восстанавливает
state tracking, не ломая обычную ассоциативную память), `mimo_rank` 2 для `tiny`
и 4 для `base`.

### 2.2 Почему гибрид, а не чистый SSM

Память чистого SSM — сводка фиксированного размера, поэтому точное извлечение
произвольного раннего токена ему даётся плохо. Все опубликованные гибриды —
Samba, Nemotron-H, Jamba, MiniMax-01 и сам Mamba-3 — тратят 8–17% слоёв на
внимание именно ради этого. У нас `attn_period = 6` (`tiny`, 4 из 24 слоёв,
17%) и `7` (`base`, 4 из 28, 14%).

Внимание — GQA (`attn_kv_heads` 2 и 4) со **скользящим окном 1024**. Samba
показал, что глобальное внимание почти ничего не добавляет, когда длинную
дистанцию держит SSM, а стоит квадратичной памяти. Окно оставляет модель
позиционно-относительной: `seq_len = 2048` — это выбор стоимости обучения, а не
жёсткий предел на инференсе.

Внимание ставится на *последний* слой периода, поэтому слой 0, читающий сырые
эмбеддинги, всегда SSM (`config::Model::mixer`).

### 2.3 Форма: глубоко и узко

`tiny` — 24 × 640, `base` — 28 × 1536. MobileLLM измерил, что ниже 1B глубина
бьёт ширину при равном числе параметров; кроме того, при нашем бюджете токенов
(§4) широкие слои остались бы недообученными.

`tiny` использует tied embeddings (эмбеддинг — 21.0M из 162.5M, 13% бюджета,
отдавать их дубликату дорого). `base` — нет: там эмбеддинг всего 4.5%, а
связанная матрица формируется предсказанием выхода в ущерб входу.

### 2.4 Мелочи, которые дёшевы и работают

| Приём | Зачем |
| --- | --- |
| SwiGLU, `ffn_mult = 2.5` | стандарт после PaLM/LLaMA; 2.5 вместо 8/3 — компенсация того, что SSM-слой сам по себе шире |
| pre-norm RMSNorm | устойчивость при глубине 24–28 |
| QK-norm во внимании | убирает всплески логитов, которые на fp32 без AMP видны как расходящийся loss |
| gated RMSNorm перед `out_proj` в SSM | держит масштаб выхода, когда MIMO суммирует несколько каналов чтения |
| `z-loss = 1e-4` | держит softmax-нормализатор около 1; страховка от дрейфа логитов на длинном прогоне |
| WSD-расписание LR | любая точка стабильной фазы — валидный чекпоинт для resume/fork, в отличие от cosine |

### 2.5 Почему **не** sparse MoE

MoE даёт ёмкость без роста FLOPs — но ценой роста числа параметров, а параметры
на 16 GB стоят 16 байт каждый (веса + градиенты + два момента Adam, §3). MoE
меняет ровно тот ресурс, которого у нас меньше всего, на тот, которого больше.
Конкретно: `base` уже занимает 16.65 GiB состояний в fp32 — любой MoE-множитель
ёмкости выкидывает модель из карты целиком.

Дополнительно: эффективный MoE требует grouped GEMM и scatter/gather по
экспертам; в burn+wgpu таких примитивов нет, наивная реализация через маски
уничтожает саму экономию FLOPs.

Поэтому ёмкость в ограниченном размере покупается иначе: MIMO (§2.1), гибрид
(§2.2), глубина (§2.3) и качество данных (§5).

## 3. Бюджет памяти

`quasar budget` печатает разбивку. Итог:

| | `tiny` | `base` |
| --- | --- | --- |
| параметры | 162.5M | 1117.5M |
| embedding / head | 21.0M / 0 (tied) | 50.3M / 50.3M |
| ssm | 63.8M | 499.3M |
| attention | 3.9M | 22.0M |
| ffn | 73.7M | 495.5M |
| fwd FLOPs/токен | 360.8M | 2306.2M |
| step FLOPs/токен (×3) | 1082.5M | 6918.6M |
| состояния bf16 (8 B/param) | 1.21 GiB | 8.33 GiB |
| состояния fp32 (16 B/param) | 2.42 GiB | **16.65 GiB** |

«Состояния» — веса, градиенты и два момента Adam, до единой активации.

**Найденная проблема.** В burn на ревизии `d028234e` нет mixed precision: ни
политики DType, ни autocast-хуков в `burn-dispatch` (единственное близкое —
`cast_to_module_dtype` в `burn-core/src/store/mod.rs`). Значит обучение идёт в
fp32, 16 B/param, и **`base` в 16 GB с AdamW не помещается** — 16.65 GiB
состояний против 16 GiB всей памяти карты.

Варианты, в порядке предпочтения:

1. **Muon вместо AdamW.** Он уже есть в burn (`burn-optim/src/optim/muon.rs`) и
   `MuonConfig::init()` возвращает тот же тип `ModuleOptimizer`, что и AdamW, —
   переключатель стоит одной строки. Хранит только момент: ~12 B/param ≈
   12.5 GiB, плюс заявленный выигрыш по эффективности на токен.
   Оговорка, которую нельзя замалчивать: документация Muon в burn прямо
   требует применять его только к 2D+ весам («Use AdamW or SGD for biases,
   embeddings, and layer norms»), а `ModuleOptimizer` применяет один оптимизатор
   ко всем параметрам. Корректная схема требует разделить `GradientsParams` по
   `ParamId` на два набора и держать два оптимизатора; API (`GradientsParams::{get,
   remove, register}`) это позволяет, но это отдельная работа, и до неё
   переключатель `--optimizer muon` был бы тихо неправильным.
2. **Activation checkpointing** (`DispatchDevice::autodiff_checkpointed`) — режет
   активации, а не состояния оптимизатора, то есть решает вторую половину задачи.
3. **Уменьшить `base`** до ~700–800M, если измеренная скорость (§4) всё равно не
   даёт обучить 1.1B до осмысленного числа токенов.

Пока это не сделано, воспроизводимо обучается `tiny`; `base` в fp32 требует
одного из трёх шагов выше. Прятать это за оптимистичной таблицей смысла нет.

## 4. Бюджет времени

Пиковая bf16-производительность RX 9070 XT — порядка 97 TFLOP/s, но burn+wgpu
без ручных ядер и без AMP реально достигает лишь части этого. **Реальную цифру
нужно измерить**, а не постулировать: harness печатает время на окно шагов, и
`tokens/s` считается из `micro_batch × accum × seq_len` за это окно.

Ниже — сколько токенов помещается в бюджет при разной достигнутой скорости
(токены = время × скорость / step FLOPs на токен):

| достигнуто | `tiny`, 24 ч | `base`, 14 дней |
| --- | --- | --- |
| 5 TFLOP/s | 0.40B | 0.87B |
| 10 TFLOP/s | 0.80B | 1.75B |
| 20 TFLOP/s | 1.60B | 3.50B |
| 40 TFLOP/s | 3.19B | 7.00B |

Chinchilla-оптимум (20 токенов на параметр) — 3.25B для `tiny` и 22B для `base`.
`tiny` при хорошей скорости в него попадает; `base` — нет ни при какой, даже за
месяц. Это не дефект плана, а факт про одну потребительскую карту, и он
определяет цели в §6: `base` заведомо будет недообучен относительно Chinchilla,
поэтому сравнение с Llama-2 3B (2T токенов) честным быть не может.

Практический вывод: **приоритет — `tiny`**, а `base` имеет смысл запускать
только после того, как измеренная скорость окажется в верхней части таблицы;
иначе те же недели лучше потратить на `tiny`, обученный далеко за Chinchilla
(SmolLM2 и Qwen показывают, что для малых моделей это работает).

## 5. Данные

Основной корпус — **FineWeb-Edu, `sample/10BT`** (parquet, ~28 GB). Причины:
образовательный фильтр даёт примерно двукратный выигрыш по downstream-качеству
на равном числе токенов относительно сырого Common Crawl, а 10BT покрывает даже
верхнюю строку таблицы §4 с запасом.

```
hf download HuggingFaceFW/fineweb-edu --repo-type dataset \
    --include "sample/10BT/*" --local-dir data/fineweb-edu
```

Опциональная примесь для `base`: 5% FineMath-4+ и 5% кода — обе улучшают
рассуждение сильнее, чем такой же объём веба. `Corpus` читает parquet/jsonl/txt
и рекурсивно обходит каталоги, так что примесь — это просто ещё один путь в
аргументах `prepare`.

Токенизатор — byte-level BPE (`tokenizers` 0.23) с разбиением цифр,
`vocab_size = 32768`. Потолок словаря — 65535: шарды хранят токены как `u16`,
что вдвое дешевле по диску и по чтению, чем `u32`, и этого достаточно.

Формат шардов: плоские `u16` + `meta.json` с `vocab_size`, числом токенов и
`bytes` исходного текста. Последнее нужно для **bits-per-byte** — единственной
метрики, сравнимой между моделями с разными токенизаторами. Token-level
perplexity сравнима только внутри одного словаря; поэтому у quark (ppl ≈ 20 на
своём токенизаторе) и у quasar общей может быть только bpb.

## 6. Цели

Цели разные по смыслу, а не только по числу: `tiny` меряется против моделей
своего класса, `base` — против того, что вообще достижимо за недели на одной
карте.

### `quasar-tiny` (162.5M, ≤ 24 ч)

| Метрика | Цель |
| --- | --- |
| bits-per-byte на holdout FineWeb-Edu | ≤ 1.00 |
| perplexity (свой 32k BPE) | ≤ 16 |
| соперники | Pythia-160M, SmolLM2-135M, quark (только по bpb) |

Смысл цели: обойти quark не за счёт размера, а за счёт архитектуры и данных, и
войти в диапазон моделей своего класса, обученных на порядки больших кластерах.

### `quasar-base` (1117.5M, недели)

| Метрика | Цель |
| --- | --- |
| bits-per-byte на holdout FineWeb-Edu | ≤ 0.85 |
| perplexity (свой 32k BPE) | ≤ 10.2 — **только** при ≥ 20 TFLOP/s и ≥ 5B токенов |
| соперники | модели того же бюджета обучения, не Llama-2 3B |

Планка ppl ≤ 10.2 из постановки задачи оставлена, но с явным условием: при
скорости из нижней половины таблицы §4 `base` получит меньше 2B токенов, и ни
одна архитектурная правка этого не компенсирует. В этом случае правильный ход —
уменьшить `base` (§3, пункт 3), а не двигать цель.

## 7. Что уже проверено

- Аналитический бюджет параметров совпадает с построенным модулем (тест).
- Полный конвейер — токенизатор, шарды, обучение, eval, генерация — прогоняется
  за минуту на CPU: `examples/smoke.sh`. Именно этот прогон вскрыл три
  реальные ошибки: несовпадение словаря шардов и конфига, коллизию финального
  чекпоинта с периодическим и деление залогированного loss на длину окна.
- Тяжёлое обучение локально **не** запускалось намеренно: harness готовится для
  запуска на пользовательской GPU.
