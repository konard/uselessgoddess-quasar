# Разбор скорости обучения

> Для максимальной скорости теперь есть отдельный upstream Triton SSD path,
> описанный в [`TRITON.md`](TRITON.md). Этот документ сохраняет расследование и
> цифры portable Burn backend; их нельзя переносить на Triton без нового
> измерения на RX 9070 XT.

Этот документ фиксирует расследование issue #7. Тяжёлый GPU-бенчмарк не
запускался: в issue прямо просили его не запускать, поэтому ниже отделены
измеренные пользователем величины от расчётов и выводов из кода.

## Что означало `40 / 60000`

Burn TUI считал optimizer steps. При `micro_batch=1`, `accum=96` и
`seq_len=2048` один такой шаг содержит

```text
1 × 96 × 2048 = 196608 токенов.
```

Измеренные 1700 tok/s дают `1700 × 3600 / 196608 = 31.1` optimizer steps в час.
Наблюдение примерно 40 шагов за первый час поэтому согласуется с throughput;
это не зависание и не ошибка счётчика. Старый план 60 000 шагов содержал 11.80B
токенов с этими batch-настройками и требовал около 80.3 суток при неизменной
скорости. С default batch он содержал ещё больше — 15.73B токенов.

## Найденные причины

1. Default recipe был завышен. Для модели с 162.5M параметров
   compute-efficient ориентир около 20 токенов на параметр равен 3.25B, но
   60 000 default-шагов задавали 15.73B. Default изменён на 12 500 шагов
   (3.2768B токенов); warmup и decay сокращены с сохранением их долей.
2. На каждом logging step цикл вызывал `into_scalar` после каждого
   micro-batch. Это синхронизирует host с device 96 раз при конфигурации из
   issue. Теперь средний loss суммируется на device и читается один раз.
3. Burn и burn-mamba в закреплённых ревизиях обучают эту модель в fp32 без
   AMP. `Mamba3SsdPath::SerialRecalculated` пересчитывает внутренности SSD ради
   памяти, а сам burn-mamba прямо выбирает переносимость backend вместо
   максимальной raw throughput. Это реальные ограничения скорости, но не
   случайный token-by-token путь: SSD обрабатывает sequence чанками. На момент
   расследования закреплённая ревизия burn-mamba совпадала с вершиной upstream,
   поэтому простого dependency update, устраняющего проблему, нет.
4. Activation checkpointing включён по умолчанию, потому что он нужен для
   ограниченной VRAM, и тоже покупает память дополнительными вычислениями.
   `--checkpointing false` можно применять только если выбранный micro-batch
   действительно помещается; это компромисс, а не универсальное ускорение.
5. Muon в запуске из issue тоже был включён. Его ортогонализация выполняется
   один раз на optimizer step — после всех 96 forward/backward micro-batches,
   а не на каждом из них. Поэтому он входит в измеренные 1700 tok/s, но не
   объясняет ни смысл `40 / 60000`, ни разницу порядков величины с quark.
   `--muon false` годится для отдельного A/B-измерения, но добавляет второй
   optimizer moment и тем самым повышает расход VRAM; заранее объявлять его
   ускорением без измерения нельзя.

## Что изменилось в наблюдаемости

Перед первым шагом harness теперь печатает число optimizer steps, полный token
budget и tokens/step. Каждое окно логирования показывает throughput, effective
TFLOP/s и ETA; те же поля видны в TUI. Поэтому невозможный wall-clock план виден
до того, как на него потрачен день.

Новый default с исходным batch — 3.2768B токенов, или примерно 22.3 суток при
1700 tok/s. С настройками из issue (`1 × 96`) 12 500 шагов — 2.4576B токенов и
около 16.7 суток. Для полного ориентира 3.25B при таком batch нужно около 16 531
шага и 22.1 суток. Заявленные ранее 24 часа достижимы только около 37 600 tok/s,
то есть примерно 40.7 effective TFLOP/s для `tiny`; наблюдалось около 1.84.

## Почему сравнение с quark вводило в заблуждение

`quark_22m` примерно в 7.4 раза меньше по параметрам, а один проход по
WikiText-103 — порядка 0.14B токенов. Старый запуск quasar планировал 11.80B
токенов, то есть более чем в 80 раз больше данных, для существенно большей и
более дорогой модели. Час на один epoch quark поэтому не предсказывает сутки на
полный run quasar. Сравнивать реализации стоит по tok/s, effective TFLOP/s и
одинаковому token budget, а качество — по bits-per-byte.
